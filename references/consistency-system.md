## 一、知识图谱一致性守护（从 vibe-noveling 移植）

### 1.1 核心原则

**"知识图谱中没有的设定 = 不存在。正文与图谱冲突 + 图谱无解释 = 问题。"**

**"禁止脑补：不能假设'可能是故意设计的''可能设定中有但没写'。"**

只有设定文件中明确写明的解释才能作为豁免依据。

**适用范围：** 角色外貌、性格、身份、能力、关系、物品属性、地理设定、历史事件、时间线等所有已建档的设定信息均受此规则约束。

### 1.2 四阶段检查流程

每章完成后执行：

**阶段 1：提取正文实体**

扫描章节，提取所有出现的人物、地点、物品、势力、设定信息。对每个实体记录：外貌描述、行为状态、与其他实体的互动关系。

**实体分类优先级：** 主角 > 重要配角 > 关键道具 > 核心地点 > 次要角色/地点 > 世界观设定细节

**阶段 2：查询知识库**

对每个提取的实体，查询项目设定文件：

| 查询维度 | 对应文件位置 |
|----------|-------------|
| 外貌/性格/身份/能力设定 | `characters/` 对应角色文件 |
| 关系设定 | `characters/` 对应角色的关系节 |
| 物品设定 | `novel-state.md`（道具追踪节） |
| 地点设定 | `world.md`（地点节） |
| 时间设定 | `novel-state.md`（时间锚节） |

**阶段 3：对比验证**

| 情况 | 判定 | 处理 |
|------|------|------|
| 正文 = 设定 | 通过 | 无需处理 |
| 正文 ≠ 设定，设定中有解释 | 通过 | 例如：掩护身份、刻意伪装 |
| 正文 ≠ 设定，设定中无解释 | **问题** | 必须修复 |
| 设定中无此实体 | **需补充** | 新实体，需创建设定 |

**阶段 4：输出报告**

```
## 一致性检查报告 - 第X章

### 通过项 (N)
- [列表]

### 问题项 (N)
- [问题描述] → [修复建议]

### 新增项 (N)
- [新实体] → [需创建的设定文件]
```

### 1.3 认知冲突专项检查

对于"角色A认为X"的认知冲突：

```
步骤1：提取正文认知 — 角色的行为/想法/言语
步骤2：对照设定事实 — 角色的真实身份/处境
步骤3：查询设定是否有解释（掩护身份、隐瞒、误会）
步骤4：有解释 → 通过（确认正文表达正确）
       无解释 → 问题（POV违规或设定矛盾）
```

示例：

```
正文（第147行）：工厂工人，满手机油味
设定：军人，第一批探索者
查询"掩护身份"：无结果
判定：问题
修复方案：A) 修改正文 B) 在设定中添加掩护身份说明
```

### 1.4 知识状态检查

> 本节内容整合自已合并的 knowledge-states-guide.md

知识状态追踪每个角色知道什么、相信什么、不知道什么。这是防止 POV 违规和生成戏剧性讽刺的主要工具——也是小说中最强大的技巧之一。

#### 1.4.1 核心规则

- 角色**不可能知道**的事情，不能出现在该角色的 POV 叙述中
- 角色**已经知道**的事情，不能表现出重新发现的惊讶
- 信息传递必须经过合理渠道（亲历、被告知、推理得出）

#### 1.4.2 三分类模型

对每个角色，维护三个列表：

**1. Knows（事实认知）**

角色作为已验证事实知道的事情。他们拥有直接证据或可信的确认。

规则：
- 只有当角色亲眼目睹、被告知、或凭借充分证据推理得出时才能添加
- 不能包含角色"应该知道"但从未展示过如何获知的事情
- 更新时机：角色目击事件、被告知可靠信息、发现证据

**2. Believes（信念认知）**

角色信以为真但可能有误的事情。包括假设、当作事实的传闻、以及善意的错误。

规则：
- 角色可以相信事实上是错误的事情
- 信念驱动行为——角色基于他们所相信的而非事实真相来行动
- 可转化为"事实认知"（如果被证实）或变得无关紧要（如果被证伪但角色未得知）
- 更新时机：角色做出假设、接受传闻、被欺骗

**3. Doesn't Know（认知盲区）**

角色**不知道**的事情，尤其是那些如果知道就会改变其行为的事情。这是戏剧张力的金矿。

规则：
- 聚焦于角色**应该关心**的事情，而非故事中的所有内容
- 优先处理具有戏剧潜力的认知盲区
- 可转化为"信念认知"（如果听到传闻）或"事实认知"（如果得知真相）
- 更新时机：新信息进入故事但角色并不知情

#### 1.4.3 角色知识状态文档格式模板

```
## [角色名称] — 知识状态

### 事实认知（Knows）
- [事实1]
- [事实2]
- ...

### 信念认知（Believes）
- [信念1]（来源：如何/为何产生此信念）
- [信念2]（来源：...）
- ...

### 认知盲区（Doesn't Know）
- [盲区1] —（影响：如果知道会发生什么变化）
- [盲区2] —（影响：...）
- ...
```

#### 1.4.4 更新规则

**每章完成后：**

1. **审查 POV 角色的知识**：他们学到了什么新东西？形成了什么新信念？
2. **审查非 POV 角色**：在他们面前发生的事件他们现在是否知道了？不在场的事件他们是否仍然不知道？
3. **检查信念修正**：角色是否发现之前的信念是错误的？将其从"信念认知"移至"事实认知"（附正确信息）或标记矛盾。

**更新时机：**
- 知识状态在**状态提取阶段**更新，而非写作过程中
- 第 N 章结束时的知识状态是写作第 N+1 章的约束条件

**检查清单：**

```
□ POV角色是否知晓了只有其他角色才知道的信息？
□ 角色是否忘记了之前已经获得的关键信息？
□ 新获得的信息是否有合理的获取途径？
□ 多个角色对同一事件的理解是否符合各自的信息获取水平？
```

#### 1.4.5 POV 违规防护

在撰写任何章节之前，检查 POV 角色的知识状态：

```
写前检查 — [角色] 作为第 [N] 章 POV：
✅ 角色已知（Knows）：[本章相关的事实]
✅ 角色相信（Believes）：[可能影响其感知的相关信念]
❌ 角色不知（Doesn't Know）：[绝不能出现在其叙述中的内容]
```

**常见违规类型：**

| 违规类型 | 示例 | 修复方法 |
|----------|------|---------|
| 全知感知 | POV 角色描述他人隐藏的情绪 | 仅展示可观察的行为；让读者自行推断 |
| 知识泄漏 | POV 角色引用从未获知的信息 | 删除或添加一个获知该信息的场景 |
| 信念当事实 | 叙述将信念当作已验证事实陈述 | 使用反映不确定性的模糊措辞 |
| 未卜先知 | 角色预判了不可能预测的事件 | 重写为仅展示角色能合理预期的内容 |

#### 1.4.6 戏剧性讽刺生成

戏剧性讽刺发生在读者知道某个角色不知道的事情时。生成方式如下：

**跨角色知识盲区：**

```
角色A 已知（Knows）：使节是刺客
角色B 不知（Doesn't Know）：使节是刺客
→ 当A和B与使节互动时，读者会感到紧张
```

**信念 vs 现实盲区：**

```
角色A 相信（Believes）：国王是仁慈的
现实（读者已知）：国王下令了屠杀
→ 角色A对国王的忠诚制造了戏剧张力
```

**在写作中的使用方法：**

1. **场景开始前**，检查所有在场角色的知识状态
2. **识别盲区**——角色知道什么与读者知道什么之间的差距
3. **撰写场景**时让盲区创造潜台词、张力或黑色幽默
4. **永远不要提前弥合盲区**——直到故事需要时才能弥合

#### 1.4.7 特殊情况

**欺骗**

当角色A对角色B说谎时：
- 角色A：已知真相，知道自己说了谎
- 角色B：相信谎言（添加到"信念认知"，来源注明"A告诉了他们"）
- 标记盲区：B 不知道真相（影响：将基于错误信息做出反应）

**部分信息**

当角色只获知了部分真相时：
- 将已确认的部分事实移入"事实认知"
- 剩余未知部分保留在"认知盲区"
- "信念认知"中可包含角色对未知部分的推测

**集体知识**

当一群人同时获知某事（如公开宣告）时：
- 同时更新所有在场角色的"事实认知"
- 注明哪些角色**不在场**（他们仍停留在"认知盲区"）

**可靠与不可靠信源**

当角色从可靠性存疑的信源获取信息时：
- 归入"信念认知"而非"事实认知"，注明信源
- 为实际真相添加一条"认知盲区"条目
- 这为后续信念被证伪留下操作空间

**时间线差异**

当角色在不同时间点获知同一事件时：
- 分别记录每个角色获知信息的确切章节和时间点
- 标记"时间差"——A在第3章知道，B在第7章才知道
- 在B获知之前的章节中，确保B的行为不反映该信息
- 常见场景：信件延迟送达、消息传播的时间差、多线叙事的时间交叉

#### 1.4.8 知识状态存储

知识状态在以下位置维护：
1. **每章快照**：每章结束时的增量快照（存储在 `state/chNNN.md`，不可变，仅记录本章变化）
2. **累计文件**：跨章节持续更新的完整知识状态（存储在 `knowledge-state.md`，可随写作过程更新）

累计文件在每章后更新。每章快照捕获增量变化（本章改变了什么）。

## 二、四表一致性系统（从 novel-writer-workflow 移植）

### 2.1 道具链表 (Prop Ledger)

追踪关键物件在每章的位置/状态/变化。仅追踪影响剧情的物品。

```markdown
### [道具名称] - [道具描述]
| 章节 | 位置 | 持有者 | 状态 | 变化说明 |
|------|------|--------|------|---------|
| Ch01 | 主角房间 | 主角 | 完好 | 首次出现 |
| Ch05 | 战场 | 主角 | 轻微损坏 | 首次使用 |
| Ch08 | ？ | ？ | ？ | 被夺走？需确认 |
```

**规则：** 状态不明用"？？？"标记，后续章节优先确认去向；所有状态变化（损坏/易手/遗失）必须有剧情交代。

**追踪判断标准：** 满足任一即需追踪——后续剧情会再次出现、与角色能力/身份相关、某个情节的关键道具、有特殊属性或能力。

### 2.2 时间锚表 (Time Anchor Table)

追踪每章的时间线和章间跨度：

```markdown
| 章节 | 时间起点 | 时间终点 | 章内跨度 | 与前章间隔 | 备注 |
|------|---------|---------|---------|-----------|------|
| Ch01 | 第1天清晨 | 第1天傍晚 | 约12小时 | - | 故事开始 |
| Ch02 | 第2天上午 | 第2天深夜 | 约16小时 | 约12小时 | 连续 |
| Ch03 | 第5天 | 第5天 | 约8小时 | 约3天 | 时间跳跃 |
```

**检查规则：**

1. **空间连续性**：前章结尾在A地、本章开头在B地时，需满足合理移动条件
2. **时间跳跃**：间隔 > 1天时，需交代期间发生的事或明确省略
3. **环境一致**：季节/天气描述与时间线对应；昼夜与时间段对应
4. **年龄追踪**：故事跨越数月/年时，备注角色年龄变化

### 2.3 术语一致性表 (Terminology Table)

统一专有名词的叫法：

```markdown
| 统一叫法 | 禁用别名 | 首次出现 | 备注 |
|---------|---------|---------|------|
| 灵力 | 灵气、灵能、精神力 | Ch01 | 力量体系的通用称呼 |
| 破境 | 突破、升级、进阶 | Ch03 | 修炼提升的术语 |
| 天机阁 | 天机楼、天机府 | Ch05 | 情报组织的正式名称 |
```

**规则：** 确定统一叫法后全文必须使用，不得混用别名。不同角色可以有不同的说话风格，但专有名词不能变。如果某个别名确实需要出现（如表现角色无知），必须是有意设计且上下文能看出是错误用法。新术语首次出现时必须在本表登记。

### 2.4 关键数字表 (Key Number Table)

追踪所有出现过的具体数字：

```markdown
| 类别 | 内容 | 章节来源 | 备注 |
|------|------|---------|------|
| 年龄 | 主角：18岁 | Ch01 | 出场年龄 |
| 年份 | 天历342年 | Ch01 | 故事发生的年份 |
| 人数 | 天机阁弟子：127人 | Ch05 | 正式弟子数 |
| 金额 | 赏金：1000灵石 | Ch08 | 任务悬赏 |
| 距离 | 两城之间：300里 | Ch10 | 地理距离 |
```

**规则：** 数字再次提及时必须与首次值一致。变化需交代原因。模糊数字（"约""近"）也需记录，避免后续用精确数字与之冲突。

## 三、上下文路由策略（从 vibe-noveling 移植）

> 操作级加载序列见 pre-writing-brief.md，本节侧重路由逻辑与优先级定义。

### 3.1 核心原则

**不全量加载记忆文件。按话题 x 动作选最小上下文包。**

### 3.2 动作类型与上下文需求

| 动作 | 需要的上下文 | 不需要 |
|------|------------|--------|
| 写新章节 | 本章大纲+前章内容+出场角色+活跃伏笔 | 世界观全部历史 |
| 修改角色 | 该角色档案+相关角色+涉及的关系 | 不相关角色的档案 |
| 修改世界观 | 被修改的设定+受影响的规则 | 未涉及的地理/历史 |
| 审查章节 | 章节正文+相关设定+时间线 | 其他章节大纲 |
| 规划未来剧情 | 全局大纲+未回收伏笔+角色弧线 | 已完成章节的详细内容 |
| 自由讨论 | 对话主题相关的设定 | 全量加载 |

### 3.3 优先级分层

> **权威定义见 SKILL.md IR-3。** 本节仅补充路由逻辑（按动作类型选择加载内容）。

P0/P1/P2 的具体内容和优先级以 SKILL.md IR-3 为准。以下为不同动作类型下的路由建议：

### 3.4 加载协议

```
1. 判定当前动作类型
2. 加载P0内容
3. 扫描P0中提到的实体，加载对应的P1内容
4. 如果P1内容引用了P2内容，按需加载
5. 总加载量控制在15000字以内
6. 如果超出，优先保留P0，压缩P1，跳过P2
```

**上下文包示例——"写第12章"：**

```
P0 加载：
  → outline.md（本章大纲部分）
  → chapters/chapter-011.md 最后3段（前章结尾）
  → novel-state.md（术语表 + 数字表）

P1 加载（根据大纲提到的角色和道具）：
  → characters/（主角文件、配角A文件）
  → novel-state.md（活跃伏笔表 + 时间锚Ch08-Ch12 + 道具追踪）

P2 按需加载（如果P1中提到）：
  → world.md（灵力体系节，如果有战斗场景）

预估总字数：约 8000-12000 字
```

## 四、章节后状态提取协议（综合最佳实践）

### 4.1 提取顺序

```
1. 保存章节正文
2. 更新主角档案（位置、状态、能力变化、新知识）
3. 更新/创建配角档案（首次出现的角色创建档案）
4. 更新伏笔追踪表（新埋设的、推进的、回收的）
5. 更新道具链表（位置/状态变化）
6. 更新时间锚表（本章时间线）
7. 更新关系网络（关系变化）
8. 更新知识状态（角色知道什么/相信什么）
9. 追加章节摘要（<=100字）
10. 更新项目状态文件
```

**执行时机：** 章节正文完成并通过一致性检查后立即执行，不要等到写完多章再补，否则容易遗漏。

### 4.2 增量更新原则

- 只更新变化的部分，不重写整个文件
- 每个更新标注章节来源
- 如果更新与已有内容冲突，先标记 `[冲突-待确认]`，询问用户

**冲突示例：**

```
已有记录：[Ch05] 主角的佩剑在黑风寨遗失
新提取信息：[Ch08] 主角手持佩剑战斗
→ 标记为：[冲突-待确认] Ch05遗失但Ch08再次出现
```

### 4.3 角色离线更新

配角在连续多章未出场后再次出现时的更新规则：

| 离线时长 | 处理方式 |
|----------|---------|
| < 5章 | 保持原样出场 |
| 5-10章 | 可有小变化（外观/心境的细微改变） |
| > 10章 | 可能有显著变化（立场转变/实力大进） |

**变化类型：** A.保持原样 B.小幅进展 C.重大突破 D.遭遇变故

**选择依据：剧情需要 > 角色目标 > 随机合理化**

即：首先考虑剧情是否需要该角色发生变化；如果剧情没有特别要求，根据角色自身的目标和性格推测其可能的变化；如果前两者都无法确定，进行合理的随机化处理。

## 五、散文基线系统（从 cw-prose-writing 移植）

> 风格预设和参数定义见 writing-styles.md，质量评估标准见 prose-quality-framework.md，本节侧重基线的存储与维护。

### 5.1 建立基线

在第一章节通过用户审核后，提取以下指标作为项目基线：

| 指标 | 计算方法 | 说明 |
|------|---------|------|
| 平均句长 | 总字数/句子数 | 句号/问号/叹号作为句子边界 |
| 句长标准差 | 各句长度与均值的标准差 | 值越大节奏感越强 |
| 短句占比 | <=10字的句子占比 | 影响紧迫感和节奏 |
| 对话比例 | 对话内容字数/总字数 | 对话括号内计为对话 |
| 段落长度均值 | 总字数/段落数 | 影响阅读疏密感 |
| 段落长度CV | 段落长度标准差/均值 | 反映段落长度多样性 |

### 5.2 基线比较

后续章节与基线比较：

| 偏差范围 | 等级 | 处理方式 |
|----------|------|---------|
| < 15% | 正常 | 允许章节间自然波动 |
| 15-25% | 关注 | 自动报告，不强制修改 |
| 25-40% | 注意 | 建议检查是否有意为之 |
| > 40% | 警告 | 强烈建议审查 |

注意：偏差不一定都是坏事。战斗章节短句更多、对话章节对话比例更高，只要服务于叙事目的即可。

### 6.2 每章标准工作流

```
1. [写前] 上下文路由 → 加载最小上下文包
2. [写中] 参考术语表和数字表 → 避免即时不一致
3. [写后] 知识图谱守护 → 四阶段一致性检查
4. [写后] 散文基线比较 → 风格一致性检查
5. [写后] 状态提取协议 → 更新所有追踪表和档案
6. [报告] 综合报告 → 一致性问题 + 风格偏差 + 状态更新
```

### 6.3 问题升级机制

| 优先级 | 问题类型 | 处理方式 |
|--------|---------|---------|
| P0-紧急 | 设定矛盾（主角身份/能力矛盾） | 立即修复 |
| P1-重要 | 数值不一致（距离/人数/时间矛盾） | 当章修复 |
| P2-一般 | 术语不一致（使用了禁用别名） | 当章修复 |
| P3-轻微 | 风格偏差（散文基线偏差 > 40%） | 通报用户决定 |
| P4-建议 | 新增实体未建档 | 下次状态提取时补充 |

## 八、伏笔账本系统与紧迫度管理

> 本节内容整合自已合并的 foreshadowing-ledger-guide.md，并融合 webnovel-writer 紧迫度公式与 InkOS Hook 生命周期管理。

### 8.1 伏笔账本概述

伏笔账本是一份持续更新的活文档，追踪小说中的每一个伏笔元素。它防止两种致命失败：

1. **遗忘的种子**：埋设了伏笔却从未回收
2. **无本之木的回报**：回报没有前期的铺垫

### 8.2 伏笔ID格式

每个伏笔项分配唯一ID：

```
FS-[章节]-[序号]
```

- **章节**：该伏笔被 PLANTED（埋设）时的章节号（3位，零填充）
- **序号**：该章节内的顺序编号（2位，零填充）

示例：
- `FS-003-01` —— 第3章埋设的第1个伏笔项
- `FS-012-04` —— 第12章埋设的第4个伏笔项

### 8.3 四状态系统

| 状态 | 含义 | 何时转换 |
|------|------|----------|
| **PLANTED（已埋设）** | 种子已植入文本。读者可能注意到也可能没注意到。 | 新伏笔元素被写入章节时 |
| **WATERED（已浇灌）** | 种子被重访或强化。读者更有可能记住它。 | 已有 PLANTED 项被引用、呼应或部分揭示时 |
| **HARVESTED（已收割）** | 回报已交付。伏笔已解决。 | 种子的完整含义或后果被揭示时 |
| **WITHERED（已枯萎）** | 种子已埋设但不会被回收。有意为之的死胡同。 | 作者决定放弃该线索（红鲱鱼、删除支线等） |

**状态转换规则：**

- PLANTED → WATERED（可发生多次）
- PLANTED → HARVESTED（回报快速到来时）
- PLANTED → WITHERED（在任何强化之前被放弃）
- WATERED → WATERED（可被多次浇灌）
- WATERED → HARVESTED（最常见的回报路径）
- WATERED → WITHERED（强化后被放弃——谨慎使用）
- HARVESTED → （终态，不再转换）
- WITHERED → （终态，不再转换）

### 8.4 账本列定义

| 列名 | 描述 | 示例 |
|------|------|------|
| **ID** | 唯一标识符 | FS-003-01 |
| **描述** | 伏笔元素是什么 | "走廊里始终显示3:15的破钟" |
| **埋设章节** | 首次植入的位置 | Ch.003 |
| **状态** | 当前状态 | PLANTED / WATERED / HARVESTED / WITHERED |
| **预期回报** | 作者打算何时/如何解决 | "Ch.018：谋杀实际发生的时间" |
| **实际回报** | 实际解决的时间和方式（收割时填写） | "Ch.019：揭示钟被撞击力停止" |
| **备注** | 额外上下文、浇灌历史、创作意图 | "Ch.007 浇灌（主角再次注意到）。Ch.012 浇灌（照片显示钟在工作）。" |

### 8.5 伏笔类型参考

| 类型 | 描述 | 示例 |
|------|------|------|
| **象征型（Symbolic）** | 后来获得意义的物品或意象 | 角色离去前反复出现的鸟的母题 |
| **对话型（Dialogic）** | 重读时获得新含义的对话台词 | "我愿意把命交给他"——说的是最终的背叛者 |
| **情境型（Situational）** | 后续呼应或反转的场景 | 一个小谎言平行对应一个毁灭性的大谎 |
| **结构型（Structural）** | 叙事结构中的模式 | 每章开头呼应前一章的结尾 |
| **角色型（Character）** | 后来变得重要的行为或特质 | 角色检查出口的习惯，在他们需要逃跑之前 |
| **世界观型（World）** | 变成剧情关键的世界观细节 | 一段看似装饰性的历史课，解释了反派的动机 |

### 8.6 写作过程中的伏笔提议格式

#### 写作章节时

在章节规划和写作过程中，如果识别到伏笔机会，使用以下格式向用户提议：

```
🔮 伏笔提议

元素：[简要描述]
埋设位置：当前章节（Ch.XXX）
预期回报：Ch.XXX 或之后
目的：[为何增强故事]
隐晦程度：[低（读者会注意到）/ 中（细心读者）/ 高（事后回顾才可见）]
```

#### 章节规划期间

检查大纲中是否有应在当前章节埋设或浇灌的伏笔项，将它们纳入章节计划。

#### 章节写作期间

如果写作中自然产生了可作为伏笔的内容（一个细节、一句对话、一个象征），标记它：

```
⚠️ 检测到伏笔候选

文本："[原文]"
潜力：[可能预示什么]
建议：是否加入账本为 FS-[XXX]-[NN]？（Y/N）
```

### 8.7 账本维护规则

1. **每章后更新**：审查章节中的新 PLANTED 项和状态转换
2. **无孤儿**：每个 PLANTED 或 WATERED 项都应有预期回报章节。如果没有，要么分配一个，要么标记为 WITHERED
3. **回报审计**：每5章审查所有 PLANTED/WATERED 项，确认它们按计划走向回报
4. **放弃需有意图**：WITHERED 项应是有意为之（红鲱鱼、删除支线），而非意外。如果某项已 PLANTED 超过10章未被浇灌，标记待决策

### 8.8 与状态提取的集成

每章后的状态提取过程必须包含：

1. 任何新埋设的伏笔项（PLANTED，附建议 ID）
2. 已有伏笔项的状态转换（WATERED/HARVESTED/WITHERED）
3. 计划中但未执行的伏笔项（标记至下一章）

### 8.9 三层伏笔紧迫度公式（从 webnovel-writer 移植）

三层伏笔及数学紧迫度计算：

- 核心伏笔（权重3.0）：目标50-300章内回收
- 副线伏笔（权重2.0）：目标30-100章内回收
- 装饰伏笔（权重1.0）：目标10-30章内回收

紧迫度公式：`Urgency = (已过章数 / 目标回收章数) × 层级权重`

状态阈值：
- CRITICAL：超过目标章数 或 核心伏笔超过50章未回收
- WARNING：超过目标的80%
- Normal：在计划范围内

同时活跃伏笔上限：推荐 ≤5条（最优），硬性上限 ≤12条（超过则强制执行"揭1埋1"规则，即每新埋一条必须先回收一条）

每5章执行紧迫度检查，更新所有活跃伏笔的状态。

## 九、Hook生命周期管理（从 InkOS 移植）

### 9.1 五种兑现时机档案

- 即时：1-3章内兑现
- 近期：3-5章内兑现
- 中程：5-10章内兑现
- 慢燃：10-20章逐步推进
- 终局：全书级别，最后兑现

### 9.2 过期升级规则

- 已加压的 hook 沉默5+章 = WARNING
- 已种植/开放的 hook 沉默10+章 = WARNING
- 核心 hook 沉默8+章 = CRITICAL

### 9.3 晋升系统

hook 被推进2+次后升级为"已晋升"，进入关键审计路径。

### 9.4 每章 Hook 账操作

每章的 Hook 账必须包含4种操作，与伏笔四态的对应关系如下：

| Hook 账操作 | 对应伏笔状态动作 | 说明 |
|------------|----------------|------|
| **Open（开启）** | → PLANTED | 本章新埋设的伏笔 |
| **Advance（推进）** | → WATERED | 本章推进/强化已有伏笔 |
| **Resolve（解决）** | → HARVESTED | 本章兑现伏笔的回报 |
| **Defer（推迟）** | 不改变状态（保持 PLANTED 或 WATERED） | 本章故意压住不动的伏笔，供写前审查使用 |

> 注意：Defer 操作不改变伏笔状态，仅标记"本章未推进"以便写前 Briefing 审查活跃伏笔。

## 十一、Claremont 系数（伏笔健康度量）

来源：dreampowers，以 Chris Claremont 命名（以撰写连载巨著中大量未解伏笔线著称）。

公式：
```
CC = 活跃伏笔数 - 已兑现伏笔数
```

解读：
- CC ≤ 0：所有伏笔已兑现。故事可能感觉"封闭"——应考虑埋设新钩子
- CC = 1-3：健康区间。未解伏笔提供了足够张力但不至于压垮读者
- CC = 4-6：警告区。读者在追踪多条开放线索——部分应尽快兑现
- CC > 6：危险区。未兑现承诺过多——读者疲劳风险。停止埋设新钩子，开始兑现

监控规则：
1. 每章结束后计算 CC
2. CC > 6 持续 3 章：严重警告——必须安排兑现章节
3. CC = 0 持续 2 章（第5章起生效）：一般警告——读者可能失去兴趣，应埋设新钩子。前4章为建章期，CC=0 不触发警告。
4. CC > 6 时不得新增钩子，除非同时兑现一条已有钩子（"揭1埋1"规则）

"揭1埋1"硬底线规则：
- 在一章中兑现 N 条伏笔时，必须至少新埋 N 条伏笔
- 防止故事耗尽张力
- 例外：终章阶段（最后10%）——允许只兑现不埋设

跨章健康追踪模板：
```
| 章节 | 新增钩子 | 已兑现 | 已推进 | 活跃总数 | CC | 状态 |
|------|---------|--------|--------|---------|-----|------|
| 1    | 3       | 0      | 0      | 3       | 3   | 正常 |
| 2    | 2       | 0      | 1      | 5       | 5   | 警告 |
| 3    | 0       | 2      | 1      | 3       | 3   | 正常 |
```

伏笔紧迫度评分和活跃上限：详见八.9（三层伏笔紧迫度公式 + 层级权重 + 状态阈值 + 活跃上限）。
