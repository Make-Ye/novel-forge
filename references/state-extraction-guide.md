# 状态提取指南

> **权威声明**: 本文件是状态提取流程的权威参考。
> 一致性系统见 consistency-system.md。状态快照模板见 templates/state-snapshot.md。

## 概述

每章写完（并可选编辑）之后，必须执行一次**状态提取**。这会记录本章中发生的一切变化，确保下一章从准确的基准现实出发。

状态提取是长篇连贯性的基石。跳过它是 AI 辅助小说写作中连续性错误的头号原因。

---

## 提取内容

### 1. 角色状态
本章中出现或有重要提及的**每个**角色：

| 字段 | 需要记录的内容 |
|------|---------------|
| **身体状态** | 当前状况（受伤、疲惫、外貌变化等） |
| **伤势追踪** | 使用下方的结构化表格记录所有伤势 |
| **情绪状态** | 章末的主要情绪 |
| **位置** | 角色在章末所处的地点 |
| **状态变化** | 社会角色、等级、声望或处境的任何变化 |
| **新技能/物品** | 本章中获得或失去的任何东西 |

#### 伤势追踪表

当任何角色有伤势时，按以下格式追踪：

```
| 部位 | 伤情 | 来源 | 当前状态 | 恢复趋势 |
|------|------|------|---------|---------|
| 左前臂 | 灼伤（光壁） | Ch8突围碰光壁 | 红了一片，未起泡 | 发痒→愈合中 |
| 右肩 | 擦伤 | Ch8跑路摔倒 | 渗血→结痂 | 发痒→愈合中 |
| 左臂（整体） | 暗河倾泻后麻木 | Ch8暗河倾泻 | 完全抬不动，脉搏极慢 | 未知 |
| 右腿 | 旧伤复发 | 前世遗留 | 抽痛，走路歪 | 慢性，不恶化 |
```

规则：
- 前几章的每处伤势**必须**在表中出现，直到明确标注为已愈合
- 每章必须更新"恢复趋势"（恶化/稳定/缓慢恢复/愈合中/已愈合）
- 如果前几章的伤势在本章中未被提及，将状态标记为"未提及"，并标记需作者确认

### 2. 关系变化
本章中互动或被讨论的每对角色：

| 字段 | 需要记录的内容 |
|------|---------------|
| **角色对** | 两个角色 |
| **变化前** | 本章之前的关系状态 |
| **变化后** | 本章之后的关系状态 |
| **催化剂** | 本章中导致变化的具体事件 |
| **走向** | 关系的发展方向（按大纲） |

### 3. 伏笔更新
本章涉及的每个伏笔条目：

| 字段 | 需要记录的内容 |
|------|---------------|
| **ID** | FS-[章节]-[序号] |
| **动作** | PLANTED / WATERED / HARVESTED / WITHERED |
| **细节** | 该伏笔条目具体发生了什么 |

**伏笔生命周期检查（每章必检）：**
- 对每个活跃伏笔，计算自上次接触（PLANTED/WATERED）以来的章数
- 15章无接触 → 标记"浇水候选"
- 30章无接触 → 标记 WARNING
- 50章无接触 → 标记 MUST_RESOLVE
- 如本章执行了 HARVEST 操作 → 必须确认已回读种植章节的 state snapshot 并验证一致性
- 在 state-snapshot.md 的伏笔操作表中记录无接触章数和生命周期状态

### 4. 知识状态
本章中每个角色，追踪其知识层面的变化：

| 类别 | 需要记录的内容 |
|------|---------------|
| **已知**（新增） | 角色学到或确认为事实的信息 |
| **信念**（新增/变化） | 角色相信但可能错误的信息 |
| **不知道**（新增盲区） | 角色应该知道但尚不知道的信息（戏剧潜力） |

### 5. 时间线
| 字段 | 需要记录的内容 |
|------|---------------|
| **章首时间** | 故事世界中本章开始的时间（日期/时刻） |
| **章末时间** | 故事世界中本章结束的时间 |
| **关键时间标记** | 本章内的任何时间跳跃 |
| **故事经过时间** | 本章覆盖的总故事内时间 |

### 6. 悬而未决的问题
本章提出但未回答的问题：
- 情节问题（X 到底在计划什么？）
- 角色问题（Y 为什么那样反应？）
- 世界问题（是什么导致了 Z 现象？）
- 读者问题（读者应该好奇的事情）

### 7. 新设定元素验证

本章引入的任何新的世界观元素、规则、角色细节或场景信息，**必须**与现有正典文件进行交叉验证。

| 字段 | 需要记录的内容 |
|------|---------------|
| **元素** | 引入的新设定元素 |
| **类别** | world-rule / character-detail / setting / ability / faction |
| **原文** | 本章中的精确引用 |
| **冲突** | 如果与 world.md 或 characters/ 中的内容矛盾，标记冲突 |
| **需要更新世界观** | YES/NO —— 是否应将该元素添加到世界观文档？ |

**验证检查清单：**
1. 将每个新元素与 `world.md` 和 `characters/` 交叉比对
2. 如果新元素与现有正典矛盾 → 标记为 CONFLICT，作者必须解决
3. 如果新元素为新但与正典一致 → 标记待添加到正典文件
4. 如果新元素在正典中已有暗示但现在明确化 → 标记为 CONFIRMED
5. 对每个新元素，判定应写入哪个文件：
   - world-rule / setting / ability / faction → world.md
   - 新主题线 / 新冲突 → plot.md
   - 计划偏差 / 涌现的事件 → outline.md（标记为"实际偏离"）
   - 重大风格变化 → prose_style.md

**涌现设定捕获（Emergent Settings Capture）：**

本章是否创造了任何 world.md/outline.md/plot.md 中原本不存在的设定？
- 是 → 记录以下信息：
  | 字段 | 内容 |
  |------|------|
  | 新设定 | 创造了什么？ |
  | 所属文件 | 应写入哪个文件？ |
  | 一致性 | 与现有正典是否一致？ |
  | 范围 | 重大（影响后续多章）还是细节（仅本章）？ |
  | 后续影响 | 是否需要在后续章节中继续展开？ |
- 否 → 记录"本章无新设定涌现"。这是正常的——不是每章都需要新发明。

**示例：**
```
| 元素 | 钱家三代人做灰市情报生意 |
| 类别 | faction |
| 原文 | Ch9: "老板姓钱。三代人做灰市的生意" |
| 冲突 | characters/ 钱鹤年条目: "灰市情报贩子" —— 一致，但需确认"三代人"是新设定 |
| 需要更新 | YES —— 在characters/钱鹤年条目补充家族背景 |
```

### 8. 读者知识审计

**为什么：** 角色知识 ≠ 读者知识。以下情况会导致"知识传递黑洞"——角色早已了解所有规则（重生者、原住民、领域专家），叙述者不许解释（POV锁 + Show Don't Tell），读者什么都没学到但系统从未检查这一点。此步骤弥合角色认知和读者认知之间的差距。

**执行步骤：**

1. **回顾本章文本**，识别读者可能从中**学到的**关于世界的新信息（角色可能早已知道）
2. **标注传达方式**：
   - 展示后果：读者通过观察某事发生后的结果来理解规则
   - 对比差异：两个不同级别/阵营角色交锋，读者从差异推断规则
   - 新手角色视角：通过一个不了解规则的角色来自然传达（对话、教学）
   - 回忆片段：老手角色回忆过去的经历，读者跟回忆一起学习
   - 对话解释：角色之间讨论规则（避免"As You Know, Bob"——不能让早已知道的人被告知）
   - 环境暗示：通过背景描写传达信息
   - 仅标签：只提到名称，没展示含义（**不够**）
3. **检查 world.md Reader Delivery Tracker** 中标注为"未传达"的项目，更新其在 state snapshot 中的状态
4. **检查前章 state snapshot 的"读者认知缺口"**，更新持续未传达章数
5. 如果有连续 ≥3 章未传达且后续情节依赖的认知缺口 → 在 state snapshot 中标记为**下一章强制传达**

**填写 state snapshot 的 Reader Knowledge Delta 部分。**

**注意：** 这一审计不改变任何角色的知识状态——角色在 §4（知识状态）中追踪。此审计只追踪**读者**的理解程度。

### 9. 读者体验审计

**为什么：** 章节级别的质量检查（POV、语域、散文）无法检测跨章节的读者体验退化。情感单调、主角长期孤立、承诺遗忘、爽点稀释、类型偏离、活动单调、世界观饿死——这些问题只有在跨章节视角下才能发现。此审计追踪八个维度的读者体验状态。

**执行步骤：**

1. **情感范围审计**
   - 识别本章主导情感类别（紧张/释放/好奇/温暖/恐惧/惊奇/满足/悲伤/幽默/愤怒）
   - 检查近5章情感分布
   - 连续 ≥3 章同情感 → WARNING；≥5 章 → MUST CHANGE

2. **互动密度审计**
   - 判定本章是否有实质性互动（对话 ≥3 轮 或 联合行动 ≥1 场景）
   - 连续 ≥3 章无互动 → WARNING；≥5 章 → MUST ADD

3. **承诺兑现审计**
   - 检查 title-synopsis.md / plot.md / outline.md 中的承诺
   - 标记本章兑现的承诺
   - 超过 10 章未兑现 → MUST DELIVER

4. **读者体验目标验证**
   - 检查 pre-writing-brief.md Segment 8 设定的体验目标
   - 判定达成/部分达成/未达成
   - 与前章目标相同 → MUST REDESIGN

5. **爽点传递审计**
   - 检查本章爽点类型
   - 更新距上次同类型/里程碑/微兑现的计数
   - 阈值判定参见 excitement-engineering.md

6. **类型承诺对齐审计**
   - 加载 title-synopsis.md，确认本书的类型承诺（主题材/副题材/核心卖点）
   - 识别本章主导叙事活动（一两个词概括本章主要在干什么）
   - 判定：该活动是否服务于类型承诺？
   - 连续 ≥3 章偏离 → WARNING；≥5 章 → MUST ALIGN

7. **活动多样性审计**
   - 识别本章主角主要活动类型（一两个词概括：逃/走/对峙/交易/探索/战斗/修炼/谈判/等待 等）
   - 检查近 5 章活动类型
   - 连续 ≥3 章同活动 → WARNING；≥5 章 → MUST VARY

8. **世界观交付审计**
   - 判定本章是否让读者获得了新的世界规则/机制认知
   - 检查近 8 章世界观交付记录
   - 连续 ≥8 章无世界观交付 → WARNING
   - 整卷无世界观交付 → CRITICAL

9. **思维留白审计**
   - 扫描本章 POV 角色的内部推理段
   - 检查是否存在完整推理链（确定的推理展示了完整过程）
   - 检查是否存在"在脑子里排了一遍"式段落
   - 存在 → 标记为 WARNING，建议修改
   - 统计：本章推理展示段数 vs 感官/行动段数，推理占比过高 → WARNING

10. **时间节奏审计**
   - 记录本章故事时间跨度（章首到章末的时间长度）
   - 检查近 5 章时间跨度分布
   - 连续 ≥5 章每章 <24 小时 → WARNING
   - 连续 ≥8 章每章 <24 小时 → MUST JUMP
   - 整卷覆盖故事时间是否合理

11. **角色出场追踪审计**
    - 更新 novel-state.md "角色出场追踪" 表：记录本章出场的每个角色的累计章数
    - 检查是否有角色第二次出场仍使用占位名（甲/乙/壮人 等）→ WARNING
    - 检查是否有角色出场 ≥3 次仍无角色档案 → WARNING
    - 检查是否有角色出场 ≥5 次仍无完整档案（仅在 supporting/ 未升级到 main/）→ MUST CREATE
    - 如果角色达到 3 次阈值：在 `characters/supporting/<name>.md` 创建简化档案（如尚未创建）
    - 如果角色达到 5 次阈值或有重要情节功能：升级到 `characters/main/<name>.md`

**填写 state snapshot 的"读者体验 Delta"部分。同时更新 novel-state.md 的"读者体验追踪"部分。**

## 提取流程

### 步骤 1：即时提取
章节文本定稿后，立即提取上述全部 9 个类别的内容。这必须在任何关于章节质量或方向的讨论**之前**完成。

### 步骤 2：交叉比对
将提取结果与以下内容比对：
- **前一章的状态快照**：验证变化是已记录的而非推测的
- **相关里程碑**：本章是否推进了卷级里程碑？
- **伏笔台账**：计划中的伏笔条目是否已执行？

### 步骤 3：格式化为快照
使用 `state-snapshot.md` 模板格式化提取结果。

### 步骤 4：更新辅助文件
快照创建后，更新以下文件：

1. **`timeline.md`** —— 为本章添加场景级条目（故事时间、持续时长、关键事件、地点、角色移动）
2. **`world.md`** —— 如果 §7（新设定元素）发现了新的且一致的元素，将其添加到 world.md 的相应章节。标记 `<!-- 更新于 Ch.XXX -->`
3. **`characters/supporting/`** —— 如果出现了新的配角，为其创建文件
4. **`characters/main/<name>.md`** —— 如果任何主角展现了新的特征/能力/关系，更新其文件
5. **`outline.md`** —— 如果本卷实际发展与原规划有重大偏差，在 outline.md 中标记偏差并说明原因
6. **`plot.md`** —— 如果涌现了新的主题线/冲突/角色弧线，添加到 plot.md
7. **`prose_style.md`** —— 如果散文风格有显著进化，更新散文基线

### 步骤 5：呈现给用户
按以下结构将快照呈现给用户：

```
## 第 [N] 章 状态提取

### 检测到的变化
[用 3-5 个要点总结关键变化]

### 详细快照
[完整格式化的快照]

### 需要确认
请审阅并确认或修正：
1. [关键点 1 —— 是否准确？]
2. [关键点 2 —— 是否准确？]
3. [是否有不一致之处？]
```

### 步骤 6：用户确认
用户可以：
- **确认**：快照作为新的基准现实保存
- **修正**：用户指出错误，快照更新后保存
- **补充**：用户添加提取遗漏的细节

## 存储约定

状态快照保存为：
```
<project-root>/state/chNNN.md  <!-- NNN = 零填充章节号 (001, 002, ...) -->
```

每章的快照会**替换**"当前状态"参考。开始新章节时，始终加载**最近**的状态快照，而不仅仅是前一章的。

### 回溯修改同步规则
编辑已写好的章节文本时，必须同步更新三个文件：

1. **章节正文** —— 章节文本本身
2. **state-snapshot.md** —— 该章的状态快照（对话列表、知识状态、伏笔、变更摘要等）
3. **timeline.md** —— 时间线中该章的对应条目

任何对章节文本的编辑都会触发三文件同步检查。读取当前的状态快照和时间线条目，与编辑后的文本比较，更新所有不一致之处。这是长篇项目中过时数据最常见的来源。

## 边缘情况

### 无变化
如果一章没有产生状态变化（罕见但可能出现在间奏章节中），仍然需要生成快照，在每个类别下明确注明"与前一次快照相比无变化"。

### 多视角章节
如果章节有多个视角段落，分别提取每个视角角色段落的内容，标注哪些观察属于哪个角色。

### 时间跳跃
如果章节包含显著的时间跳跃，且跳跃期间发生了变化，则为跳跃前和跳跃后的状态分别生成快照。

### 模糊变化
如果某个变化不明确（例如"角色可能已经知道了 X"），显式标记：
```
⚠️ 模糊：尚不清楚 [角色] 是否在此次场景中得知了 [事实]。
需要用户决定。
```
