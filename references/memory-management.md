# Novel-Forge 记忆管理系统

> **权威声明**: 本文件是记忆管理和会话恢复的权威参考。
> 上下文加载协议以 SKILL.md IR-3 为准。

## 核心原则

**文件即记忆。所有状态存储在人类可读的 Markdown 文件中。**
- 不使用数据库、JSON 结构体、或任何需要解析器才能理解的格式
- 用户可以在任何时候直接编辑任何文件
- AI 在每次读取时检测是否有手动修改，如有差异主动询问

## 一、项目目录结构

**权威定义在 SKILL.md 的 "File Structure Reference" 节。** 本节补充加载策略。

```
<project-dir>/                      ← SKILL.md 定义的结构（权威）
  novel-state.md          # Phase tracker, session history, chapter summaries, active foreshadowing
  project-config.md       # Genre, tone, length, audience, constraints
  title-synopsis.md       # Title, synopsis, metadata
  world.md                # World rules — minimum viable, grows with content (baseline)
  characters/             # Character directory
    main/                 # Main characters: detailed 3-layer profiles (3-5 per story)
      <name>.md           # One file per main character
    supporting/           # Supporting/minor characters: brief profiles
      <name>.md           # One file per supporting character (created as they appear)
  plot.md                 # Structure, central conflict, turning points, themes
  outline.md              # Volumes + milestones (NOT per-chapter detail)
  timeline.md             # Scene-level time tracking (updated after each chapter)
  prose_style.md          # [已合并入 novel-state.md 散文基线] — 参见 novel-state.md 量化指标 + 基线段落 + 风格特征摘要
  chapters/
    chapter-001.md        # Full chapter text
    chapter-002.md
    ...
  state/
    ch001.md              # State snapshot AFTER chapter 1 (immutable)
    ch002.md              # State snapshot AFTER chapter 2 (immutable)
    ...
```

**核心设计原则：**
1. `state/chNNN.md` 是不可变的。所有每章的变化量都写入 state 快照。（NNN 为零填充：001, 002, ...）
2. `characters/` 是独立目录。主角色有详细档案，配角在 `characters/supporting/` 中各一个文件。
3. `world.md` 是最小可行设定。新设定在Phase 8中从内容提取后增量补充，不预先铺满。
4. `outline.md` 使用里程碑制，不是逐章脚本。章节之间的空间在Phase 7讨论中填充。
5. `timeline.md` 追踪场景级时间精度，每章更新。

### 文件用途速查表

| 文件 | 内容 | 更新频率 |
|------|------|----------|
| `novel-state.md` | 当前阶段、最后修改时间、待办事项、焦点指引 | 每次会话开始时 |
| `project-config.md` | 类型、基调、长度、受众、约束 | 项目初始时 |
| `title-synopsis.md` | 书名、一句话概括、简介 | Phase 2 初始，Phase 6 精炼 |
| `world.md` | 世界规则（最小可行，增量生长） | 写作中发现新设定时（Phase 8） |
| `characters/main/` | 主角详细档案（3层+声音） | 角色经历重大变化时 |
| `characters/supporting/` | 配角独立档案 | 新配角出现时 |
| `plot.md` | 结构、核心冲突、转折点、主题 | 剧情架构调整时 |
| `outline.md` | 卷级方向 + 里程碑 | 里程碑调整时 |
| `timeline.md` | 场景级时间线 | 每章完成时（Phase 8） |
| `chapters/` | 已完成章节正文 | 每章完成时 |
| `state/` | 不可变章节状态快照 | 每章完成时（Phase 8） |

## 二、三层记忆模型（精简版）

三层记忆本质是**加载优先级**的分层，不是学术分类。记住以下10行即可：

```
1. 语义记忆 = 永久事实（world.md规则、characters/角色档案、已确认决策）→ 每次会话必加载
2. 情节记忆 = 历史记录（已完成章节摘要、关键场景、角色状态变化）→ 回顾情节时加载
3. 工作记忆 = 当前上下文（正在写的章节 ± 1章摘要、出场角色、活跃伏笔）→ 每章构建
4. 冲突优先级：语义 > 情节 > 工作记忆
5. 违反语义记忆的内容 → 立即暂停并报告
6. 工作记忆容量上限：15000字（超出时精简早期章节为摘要）
7. 情节记忆按章节摘要存储在 novel-state.md 的会话历史中
8. 语义记忆存储在 world.md、characters/、novel-state.md 中
9. 三层之间不重复存储，靠文件路径引用关联
10. 任何信息只存在于对话历史中 = 不存在。必须写入文件。
```

## 三、加载决策树

> 这是本文件的核心。操作级加载序列（P0→P1→P2）的完整版见 pre-writing-brief.md。

### 按项目阶段的加载决策

```
你是哪个阶段？──→ 加载这些文件
│
├─ 构思期（尚未写正文）
│   ├ 必读：novel-state.md, title-synopsis.md
│   ├ 按需：world.md, characters/
│   └ 跳过：chapters/
│
├─ 大纲期（正在规划章节结构）
│   ├ 必读：novel-state.md, title-synopsis.md, outline.md
│   ├ 按需：world.md, characters/, plot.md
│   └ 跳过：chapters/正文
│
├─ 写作期（正在写章节正文）
│   ├ 必读（P0）：novel-state.md, title-synopsis.md, 前章 state 文件（state/chNNN.md）
│   ├ 写前加载（P1）：outline.md, characters/（出场角色文件）, novel-state.md（活跃伏笔）, 前章最后500字原文
│   ├ 写中按需（P2）：world.md（地点/势力/术语节）, novel-state.md（散文基线节）
│   └ 仅精确引用时：早期章节 state 文件, chapters/ 中的特定段落（100-200字）
│
├─ 修订期（修改已完成章节）
│   ├ 必读：novel-state.md, 要修改的章节正文, outline.md
│   ├ 按需：characters/, world.md
│   └ 警惕：修改早期章节时必须检查对后续章节的连锁影响
│
└─ 会话恢复（中断后回来）
    └→ 见下方"会话恢复协议"
```

### 按任务类型的加载决策

```
你现在要做什么？──→ 额外加载
│
├─ 写对话场景 → characters/main/<name>.md 或 characters/supporting/<name>.md（声音档案 + 出场角色）
├─ 写战斗/动作 → world.md（规则/能力限制节）
├─ 写感情戏 → characters/main/<name>.md（关系节 + 出场角色）
├─ 揭示伏笔 → novel-state.md（活跃伏笔）+ 伏笔埋设章节摘要
├─ 引入新角色 → 在 characters/main/ 创建新文件（主角色）或 characters/supporting/ 创建新文件（配角）
├─ 引入新地点 → world.md（地点节）+ 一致性检查
├─ 跨章节连续性检查 → 前后各1章摘要 + novel-state.md
└─ 确认术语 → world.md（术语节）
```

### Token 预算紧张时的最低加载

> **P0/P1/P2 权威定义见 SKILL.md IR-3。** 以下为非写作场景的最低加载建议。

```
非写作场景最低（3个文件，覆盖90%需求）：
  1. novel-state.md       → 当前状态 + 已确认决策 + 活跃伏笔
  2. title-synopsis.md    → 核心设定
  3. world.md             → 硬规则

如只能再加2个（覆盖95%需求）：
  4. outline.md           → 整体结构
  5. 前章摘要              → 衔接
```

## 四、会话恢复协议

当中断后重新打开项目（无论是同一天还是数周后），按以下顺序操作：

### 恢复步骤（严格按序执行）

```
步骤1：定位当前状态（30秒）
  → 读取 novel-state.md
  → 确认：阶段、当前焦点、已完成章节数、待处理事项

步骤2：回忆上次工作内容（1分钟）
  → 读取 novel-state.md 的会话历史（最后2条记录）
  → 确认：上次做了什么、遗留问题是什么、下次计划是什么

步骤3：回忆关键决策（1分钟）
  → 读取 novel-state.md 的决策记录（最后5条）
  → 确认：最近做了哪些创意决策，避免无意识推翻

步骤4：加载当前任务所需上下文
  → 根据上方的"加载决策树"，按当前阶段加载必要文件
  → 如果正在写某章：额外加载该章的前章摘要 + characters/（出场角色文件）

步骤5：向用户确认
  → "上次我们 [做了什么]，当前计划 [做什么]，是继续还是调整？"
  → 等用户确认后再开始工作
```

### 中断时间不同的恢复策略

| 中断时长 | 额外操作 |
|----------|----------|
| 同一天内（< 8小时） | 步骤1-3即可，步骤4按需 |
| 1-3天 | 完整执行步骤1-5，重读前章摘要+最后500字 |
| 1-4周 | 完整执行步骤1-5，重读最近3章的摘要 + outline.md |
| 超过1个月 | 完整执行步骤1-5 + 通读 title-synopsis.md 和 outline.md |

### 恢复时的安全检查

```
恢复后第一件事写之前，快速检查：
□ manifest 中的"当前焦点"是否仍然准确
□ 最近完成的章节是否与 outline 一致
□ 是否有用户手动编辑过的文件（内容哈希对比）
□ 是否有超过阈值的未推进伏笔（检查 novel-state.md 活跃伏笔表）
```

## 五、FAIL/PASS 示例

### FAIL：加载所有文件每次会话

```
场景：写作第15章，用户启动新会话
错误做法：依次读取了 world.md全部, characters/全部文件,
  plot.md, outline.md全部, ch001 到 ch14 全部正文,
  novel-state.md, title-synopsis.md
后果：消耗大量 token，其中大部分信息与第15章无关
      AI 被无关信息干扰，可能出现"信息过载"导致的幻觉
诊断：没有按阶段/任务做选择性加载
```

### PASS：按需精准加载

```
场景：写作第15章，用户启动新会话
正确做法：
  P0（必读）：novel-state.md, title-synopsis.md, world.md（规则节）
  P1（写前）：outline.md, ch14摘要+最后500字, characters/（出场角色文件）, novel-state.md（活跃伏笔）
  P2（写中按需）：world.md（地点节，因为第15章在新地点），characters/（声音档案节，因为有大段对话）
  跳过：ch01-ch13正文, world.md中与本章无关的势力/历史节
后果：token 消耗合理，上下文精准聚焦当前任务
```

### FAIL：只靠对话历史记忆项目状态

```
场景：用户关闭了之前的对话，开了新窗口
错误做法：假设"上次我们讨论了X，所以当前应该在做Y"
后果：用户在旧会话外做了修改（手动编辑了文件、推翻了之前的决定），
      AI 基于过时的对话记忆做决策，与实际项目状态脱节
诊断：违反"所有状态通过文件传递"原则
```

### PASS：从文件重建上下文

```
场景：用户关闭了之前的对话，开了新窗口
正确做法：
  执行"会话恢复协议"步骤1-5
  从 novel-state.md 获取当前真实状态 + 最近决策
  从 novel-state.md 会话历史了解最近工作
后果：即使完全不知道之前的对话内容，也能准确接续
```

### FAIL：批量更新所有记忆文件

```
场景：完成了第10章
错误做法：写完正后一次性打开7个文件，批量修改所有相关内容
后果：中途某个更新失败或被打断，导致部分文件已更新、部分未更新，
      状态不一致且难以排查哪个文件是"对的"
诊断：违反"确认后立即写入，绝不批量更新"原则
```

### PASS：即时逐步更新

```
场景：完成了第10章
正确做法：
  1. 确认第10章正文 → 写入 chapters/chapter-010.md
  2. 确认伏笔状态变化 → 更新 novel-state.md 活跃伏笔表
  3. 确认角色状态变化 → 更新 characters/ 对应角色文件
  4. 更新 outline.md 标记该章完成
  5. 最后更新 novel-state.md（会话历史+当前焦点）
后果：每一步都有确认，中断后也能知道哪些已更新、哪些未更新
```

## 六、持久化协议

### 核心规则：确认后立即写入，绝不批量更新

### 每种变更的写入规范

#### 世界观变更
```
触发：用户确认新的世界观设定
写入：world.md 对应的节
格式：在相关节末尾追加或修改
标注：添加 <!-- 更新于 第X章 --> 注释
```

#### 角色状态变更
```
触发：角色发生重要变化（新技能、关系变化、心理转变）
写入：characters/ 对应角色文件的"当前状态"部分
格式：旧状态用 ~~删除线~~ 标记，新状态加粗
同步：如果影响关系，同步更新对应角色文件的关系节
```

#### 情节进展
```
触发：章节完成
写入（按顺序）：
  1. chapters/chapter-NNN.md → 章节正文
  2. state/chNNN.md → 创建该章的不可变状态快照（Phase 8 核心输出）
  3. novel-state.md → 更新章节摘要表 + 活跃伏笔摘要 + 角色状态摘要 + current_chapter + 会话历史
  4. outline.md → 标记该章为"已完成"
注意：characters/ 和 world.md 不在此时更新——它们是基准文件，每章的变化量记录在 state 快照中
```

#### 创意决策
```
触发：通过 Proposal 系统确认的决策
写入：novel-state.md（决策记录节）
格式：
  ## [日期] - [决策主题]
  - 选择：[方案名称]
  - 理由：[用户给出的理由 或 AI推荐理由]
  - 影响：[受影响的文件和设定]
```

### 绝不丢失原则

> **如果一条信息在对话中出现且被确认，它必须出现在至少一个记忆文件中。**
> 如果 AI 不确定某条信息是否已记录，宁可重复记录也不要遗漏。

## 七、人类可编辑性

### 用户编辑检测协议

AI 在读取任何记忆文件时，执行以下检查：

```
1. 内容变化检测
   - 如果文件有修改：
     → "我注意到 [文件名] 有更新。以下是我发现的变更：
        [列出差异]
        这些变更是你做的吗？我需要更新相关的记忆吗？"

2. 内部一致性检查
   - 角色 profile 状态 vs 最新章节描写
   - 时间线 vs 章节事件序列
   - 规则设定 vs 实际使用
   - 发现不一致时：
     → "我发现一个可能的矛盾：
        [文件A] 中说 [X]
        [文件B] 中说 [Y]
        哪个是最新的？"

3. 结构完整性检查
   - 所有文件是否存在，必填字段是否填写
   - 有缺失时：
     → "以下文件/信息缺失，是否需要补充：[列表]"
```

### 用户直接编辑的注意事项

- **编辑世界观文件**：AI 在下次加载时识别并适应新设定，同时检查与已有章节的矛盾
- **编辑章节正文**：AI 记录编辑，后续章节以修改后版本为正典
- **编辑大纲**：AI 重新评估后续章节计划，有冲突时提出警告
- **删除内容**：AI 不自动恢复被删除的内容，但会询问是否是有意的

## 九、记忆文件写作规范

### 通用规则

- 使用 `##` 标记主要章节，`###` 标记子章节
- 列表项用 `-` 开头，重要信息用 `**加粗**`
- 注释用 `<!-- HTML注释 -->`，日期格式：`YYYY-MM-DD`
- 章节引用格式：`第X章` 或 `chNNN`（零填充：001, 002, ...）
- 文件之间通过名称引用，不用ID或索引

### 角色 Profile（在 characters/ 目录中每人一个文件）

**主角色** — `characters/main/<name>.md`，使用 templates/character-card.md 模板，包含完整的3层模型+声音档案。
（如 `characters/main/sutu.md`）

**配角** — `characters/supporting/<name>.md`，每个配角一个文件，包含：身份、首次出现、已知信息、当前状态、叙事功能。
（如 `characters/supporting/clothshoes.md`）

配角出现≥3次或获得重要剧情功能时，从 supporting/ 移到 main/。

### 世界规则模板

```markdown
# [规则类别]

## 规则 #[编号]：[规则名称]
- **描述：** [一句话说明]
- **机制：** [详细的工作方式]
- **限制：** [不能做什么]
- **代价：** [使用/违反的代价]
- **已知例外：** [如有]
- **首次出现：** 第X章
```

## 十、会话管理

### 会话启动流程

```
AI 启动时执行：
1. 读取 novel-state.md → 了解当前阶段
2. 根据 state 中的"当前焦点"确定本会话目标
3. 按加载决策树读取必要文件（见第三节）
4. 向用户确认："上次我们 [做了什么]，今天计划 [做什么]，对吗？"
5. 确认后开始工作
```

### 会话结束流程

```
AI 结束时执行：
1. 更新 novel-state.md 的"当前状态"和"会话历史"
2. 在 novel-state.md 添加会话记录：
   | 日期 | 阶段 | 完成事项 |
   | [date] | [phase] | [摘要] |
3. 确认所有变更已写入对应文件
4. 向用户汇报："本次会话完成了 [X]，更新了 [Y] 个文件，下次建议从 [Z] 开始。"
```

### 会话间连续性保证

> **AI 不依赖对话历史来记忆任何项目信息。所有项目状态必须通过文件传递。**
> 这意味着即使切换到全新的对话窗口，只要文件在，项目就能无缝继续。

## 十一、长篇归档策略（200章以上）

> 当小说规模超过200章时，文件数量和体积会显著增长。本节提供分阶段归档与压缩策略，确保系统在长篇写作中保持高效、准确。

### 11.1 状态快照归档

state 目录中的每章快照是项目最密集的记忆文件。长篇中必须按阶段压缩早期快照。

#### 归档节奏

| 节点 | 操作 | 保留内容 | 删除内容 |
|------|------|---------|---------|
| **50章后** | 前20章的 state 文件压缩为摘要 | Delta Summary + 角色状态变化 + 关键伏笔 | 详细的事件列表、场景描述、对话摘要 |
| **100章后** | 前50章的摘要进一步压缩为卷级摘要 | 卷级核心事件 + 重大角色变化 + 跨卷伏笔 | 单章级别的所有细节 |
| **200章后** | 每50章执行一次滚动归档 | 最近50章详细 + 更早章节的卷级摘要 | 50章以前的所有单章快照 |

#### 归档操作规范

- 归档的 state 文件移至 `state/archive/` 目录
- 压缩后的摘要文件命名格式：`state/archive/vol1-ch001-020-summary.md`
- 每个归档文件不超过 **500字**
- 归档前必须确认：该范围内的所有伏笔是否已回收（未回收的伏笔必须在摘要中保留）

#### 归档文件模板

```markdown
# 卷1（第1-20章）归档摘要

## 核心事件
- [3-5个改变故事方向的关键事件]

## 角色状态变化
- [角色名]：[起始状态] → [结束状态]

## 活跃伏笔
- [仍未回收的伏笔，注明埋设章节]

## 已回收伏笔
- [已回收的伏笔，注明埋设与回收章节]
```

### 11.2 novel-state.md 大小管理

novel-state.md 是项目的核心索引，但随着章节增加会持续膨胀。

#### 触发阈值与压缩规则

- **超过8000字时触发压缩**
- 章节摘要表：保留最近 **30章** 的详细摘要，更早的合并为卷摘要
- 活跃伏笔摘要：仅保留 **未回收** 的伏笔，已回收的移至 `foreshadowing-ledger.md` 的历史区
- 角色状态摘要：仅保留 **活跃角色**（最近20章内出场），其余移至角色文件

#### 压缩后的 novel-state.md 结构

```markdown
# novel-state.md

## 当前状态
[不变]

## 卷摘要（历史章节压缩版）
### 卷1（第1-25章）
[一段话概括]
### 卷2（第26-50章）
[一段话概括]

## 章节摘要（详细，仅最近30章）
| 章节 | 标题 | 摘要 | 关键变化 |
|------|------|------|---------|
| ch171 | ... | ... | ... |

## 活跃伏笔
[仅未回收的]

## 活跃角色状态
[仅最近20章内出场的角色]

## 决策记录
[保留最近20条，更早的移至 archive/decisions.md]

## 会话历史
[保留最近10条]
```

#### foreshadowing-ledger.md（新增文件）

当活跃伏笔数量超过30条时，创建独立的伏笔总账文件：

```
<project-dir>/
  foreshadowing-ledger.md     # 伏笔总账
    ## 活跃伏笔               # 未回收
    ## 已回收伏笔             # 按卷分组归档
    ## 废弃伏笔               # 计划中但最终未使用的
```

### 11.3 角色文件管理

长篇小说中角色数量可能超过50个。需要按活跃状态分类管理。

#### 角色分类标准

| 分类 | 定义 | 存储位置 | 加载策略 |
|------|------|---------|---------|
| **活跃角色** | 最近20章内出场 | `characters/main/` 或 `characters/supporting/` | 正常加载 |
| **休眠角色** | 连续20+章未出场，但可能在后续回归 | `characters/dormant/` | 需要时按需加载 |
| **已退场角色** | 已死亡或永久离开故事 | `characters/departed/` | 仅在闪回/提及/其他角色回忆时加载 |

#### 角色归档操作

```
角色归档触发条件：
  - 休眠：连续20章未出场 → 移至 characters/dormant/
  - 退场：明确死亡/永久离开 → 移至 characters/departed/

归档时保留：
  - 完整角色档案（不移至dormant/departed时删除内容）
  - 在关系网络中的引用（不删除，标注为"[休眠]"或"[已退场]"）

新角色创建前：
  - 先检查 characters/dormant/ 中是否有可复用的角色
  - 复用休眠角色优于创建全新角色（已有关系网络和读者认知基础）
```

#### 角色索引文件

50+角色时，在 `characters/` 目录下创建 `index.md`：

```markdown
# 角色索引

## 活跃角色（最近20章出场）
| 角色 | 类型 | 当前位置/状态 | 最近出场 |

| 苏简 | 主角 | 海城·公安局 | ch198 |

## 休眠角色
| 角色 | 类型 | 休眠前状态 | 最后出场 | 回归可能性 |
|------|------|-----------|---------|-----------|
| 老陈 | 配角 | 住院中 | ch156 | 高 |

## 已退场角色
| 角色 | 退场方式 | 退场章节 | 关系网络影响 |
|------|---------|---------|------------|

```

### 11.4 timeline.md 管理

长篇小说的时间线文件会持续增长，超过500行时需要按卷分拆。

#### 分拆规则

- 超过 **500行** 时按卷分拆
- 文件命名格式：`timeline-vol1.md`、`timeline-vol2.md`
- 保留一个 `timeline-current.md` 指向当前卷
- 每个归档的旧卷时间线在文件开头保留 **卷级事件摘要**（不超过200字）

#### 分拆后的目录结构

```
<project-dir>/
  timeline-current.md      # 当前卷的详细时间线（写作时加载）
  timeline-vol1.md         # 卷1时间线（归档，按需加载）
  timeline-vol2.md         # 卷2时间线（归档，按需加载）
  timeline-vol3.md         # 卷3时间线（归档，按需加载）
```

#### timeline-current.md 头部格式

```markdown
# 时间线 - 当前卷（卷4）

> 上一卷时间线：timeline-vol3.md
> 下一卷时间线：（尚未开始）

## 卷级事件摘要
[本卷的核心事件，200字以内]

## 详细时间线
[场景级精度]
```

### 11.5 搜索优化

当项目文件总大小超过100KB时，需要采用索引驱动的搜索策略，避免逐文件读取。

#### 策略：novel-state.md 作为主索引

- 当项目文件总大小超过 **100KB** 时，将 `novel-state.md` 作为文件索引
- 读取任何文件前，先检查 `novel-state.md` 的文件清单确认路径
- 使用 **Grep** 搜索历史信息，而非 Read 逐文件读取

#### 搜索优先级

```
搜索历史信息的优先级：
  1. novel-state.md 的章节摘要表     → 快速定位相关章节
  2. Grep 关键词搜索 state/ 和 chapters/ → 精确查找具体信息
  3. characters/index.md              → 角色相关查询
  4. foreshadowing-ledger.md          → 伏笔相关查询
  5. Read 具体文件                    → 仅在已精确定位后使用
```

#### 加载优化总结

| 项目规模 | 文件总量 | novel-state.md 大小 | 加载策略 |
|---------|---------|---------------------|---------|
| 短篇（<50章） | < 30个 | < 3000字 | 全量加载关键文件 |
| 中篇（50-200章） | 30-120个 | 3000-8000字 | 按需加载 + novel-state 索引 |
| 长篇（200+章） | 120+个 | 8000字+ | 索引驱动 + Grep搜索 + 归档管理 |

**长篇核心原则：** 文件可以很多，但每次加载的文件必须精准。用 novel-state.md 做"入口"，用 Grep 做"搜索"，用 Read 做"精读"。永远不要逐文件扫描。
